# Introduction

# Compiler vs Interpreter

Compiler

A program that can read one one language - the source language - and translate it into an equivalent program in another language - the target language.

  • an offline step before execution

  • target language could be:

    1. an executable file
    2. intermediate format
    3. another programming language

Interpreter

A program that read an executable input program and produces the results of that running program.

  • runs every time the program is executed.

# Compilation Modules

Each module produces output which serves as input to the next modules.

  1. Lexical Analyzer > token stream
  2. Syntax Analyzer > syntax tree
  3. Semantic Analyzer > syntax tree
  4. Intermediate Code Generator > intermediate representation
  5. Code Optimizer > intermediate representation
  6. Code Generator > target machine code
  7. Code Optimizer > target machine code

# Qualities of a Good Compiler

Qualities of a Good Compiler

  • Produces correct code
  • Compiler & it's output runs fast
  • Output is small
  • Compile time is proportional to the program size
  • Consistent, predictable optimization
  • Good debugging & diagnostic features

# Lexical Analysis

Lexical analysis is the process of reading a stream of characters and breaking them up into lexemes.

# Definitions

Lexeme - a component substring

Lexeme

a component substring

  • each has a token/class, e.g.) identifiers, integers, operators, etc.
  • forms a token with it's token type, e.g.) <int, "0">

Regular Expressions

  • provide a formalized way of specifying the rules for describing a programming language.
  • used to define lexeme patterns in order to determine valid lexemes.
  • define all the possible strings in a particular lexeme token type.

Some more definitions

  • alphabet - a finite set of symbols.
  • string - a finite sequence of symbols drawn from an alphabet.
  • character - a string with length 1.
  • language - any set of strings over some fixed alphabet.
  • regular set/language - a language that can defined using regular expressions.
  • epsilon - empty string with length 0.

# Operations on Languages

Union - AUB

{ e | e is an element of A or e is an element of B }

Concatenation - AB

{ ab | a is an element of A and b is an element of B }

Kleene Closure - A*

if A = { A }, then A* = { E, a, aa, aaa..}, where E = empty string

Positive Closure - A+

A+ = A.A*

if A = { A }, then A* = { a, aa, aaa..}

# Finite Automata

Finite Automata

Transition graphs that recognize patterns.

  • used for the implementation of a language.

  • can be used to express any regular language.

  • recognizes a string if there is some path in it's transition graph from the start state to an accepting state such that the symbols along the path spell out the string.

  • not recognized if:

    • does not end in an accepting state.
    • gets stuck at a state with no appropritate transition.

# Types

Nondeterministic Finite Automata (NFA)

  • can have multiple transitions for the same input in a given state.
  • can have E moves.
  • slower but smaller.

Deterministic Finite Automata (DFA)

  • can have only one transition per input per state.
  • no E moves.
  • faster but bigger.

# Parsing

Parser

Checks whether a program confirms to a set of grammatical rules.

  • Accepts input tokens for the lexical analyzer, and outputs a syntax tree as input for semantic analysis.

# Context Free Grammers

CFGs specify programming language syntax constructs.

Requirements

  • Set of TerminalsT

    • define basic strings which make up the program - token types.

  • Set of Non-TerminalsN

    • define more complex language constructs.

  • A Start SymbolS

    • a non-terminal which is used a starting point to generate a complete program.

  • A Set of Predictions

    • specify how all the non-terminals are generated with terminals or other non-terminal to form more complex constructs.

# Derivation

  • a sequence of one or more productions rules.
  • deriving a program always starts at S.

Types of Derivation

  • Leftmost

    • used in top down parsing.
    • leftmost non-terminal is always chosen for expansion.

  • Rightmost

    • used in bottom up parsing.
    • rightmost non-terminal is always chosen for expansion.

Syntax Tree

Represent a derivation.

  • AKA Parse Tree

  • an in-order traversal produces the original output (TODO input?) .

  • Components:

    • terminals at the leaves, corresponding to tokens.
    • non-terminals at the interior nodes.
